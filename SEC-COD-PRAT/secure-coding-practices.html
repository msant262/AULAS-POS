Secure Coding Practices para SDLC com IA assistida

IA já faz parte do desenvolvimento. O problema não é usar IA para codar; o problema é usar IA sem threat model, revisão, testes, rastreabilidade, sandbox, permissões e gates de segurança.

01 Estado da arte

Dados para calibrar a conversa

A leitura central: IA acelera criação de código, mas também desloca o gargalo para validação, governança e feedback loops.

Discussão

O número “80%+” depende do denominador

Stack Overflow fala em 84% usando ou planejando usar IA; JetBrains aponta 85% usando regularmente; Sonar observa 72% de uso diário entre quem já testou IA. O material explicita a fonte para evitar generalização fraca.

Gargalo

Review e validação viram o centro

GitLab reporta que 85% veem o gargalo migrar para review/validação. Sonar coloca “revisar e validar código gerado por IA para qualidade e segurança” como habilidade principal na era de IA.

Risco

Funcional não significa seguro

O relatório da Veracode encontrou 45% de amostras falhando testes de segurança. A aula trata todo output de IA como proposta, não como evidência.

Stack Overflow 2025, JetBrains 2025, GitHub Octoverse 2025

Adoção de IA no desenvolvimento

IA deixou de ser exceção. O controle de segurança precisa assumir que código, testes, docs e reviews podem ter participação de IA.

Sonar State of Code Developer Survey 2026

Código gerado ou significativamente assistido por IA

O volume cresce mais rapido que os processos tradicionais de revisão. Batch pequeno, CI rapido e abuse tests passam a ser controles centrais.

Veracode GenAI Code Security Report 2025

Código gerado por IA: passou vs falhou em segurança

Funcional e sintaticamente correto não significa seguro. Revisão precisa procurar classes conhecidas de falha.

Veracode 2025

Falhas por linguagem em tarefas de IA

O risco varia por linguagem e padrão, mas nenhuma stack dispensa validação.

GitLab AI Accountability Report 2026

Governança de código gerado por IA

O desafio deixa de ser só gerar código e passa a ser saber origem, intenção, responsabilidade e risco residual.

arXiv 2406.10279

Package hallucination e slopsquatting

Dependência sugerida por IA deve ser tratada como input não confiável até haver verificação externa.

DORA Impact of Generative AI

IA amplifica o sistema de entrega existente

Sem feedback loop forte, mais geração pode significar mais batch, review lento e instabilidade.

TypeScript/JavaScript

SQL injection e XSS no mesmo fluxo

gate: Unit test com payload de injection
Vulnerável
app.get('/users', async (req, res) => {
  const q = req.query.q || '';
  const rows = await db.query("select * from users where name like '%" + q + "%'");
  res.send('<ul>' + rows.map(u => '<li>' + u.name + '</li>').join('') + '</ul>');
});
Seguro
const schema = z.object({ q: z.string().max(80).optional() });

app.get('/users', async (req, res) => {
  const { q = '' } = schema.parse(req.query);
  const rows = await db.query('select * from users where name like ?', ['%' + q + '%']);
  res.render('users', { users: rows }); // template auto-escapes HTML
});

Risco: Concatena input em SQL e HTML. Um payload controla consulta e renderização.

Gate: Unit test com payload de injection, SAST e revisão de output encoding.

Prompt fraco: Faça busca por nome com SQL simples e renderize HTML.

Prompt melhor: Implemente busca com schema validation, prepared statement, template auto-escape e testes negativos de SQLi/XSS.

Python

Path traversal em download

gate: Teste com ../
Vulnerável
@app.get('/download')
def download():
    name = request.args['file']
    return send_file('/srv/reports/' + name)
Seguro
BASE = Path('/srv/reports').resolve()

@app.get('/download')
def download():
    name = ReportQuery.model_validate(request.args).file
    target = (BASE / name).resolve()
    if BASE not in target.parents:
        abort(404)
    return send_file(target)

Risco: Nome vindo do cliente pode usar ../ para ler arquivo fora da pasta esperada.

Gate: Teste com ../, allowlist de extensão e revisão de path normalization.

Prompt fraco: Crie endpoint para baixar qualquer arquivo enviado no parâmetro.

Prompt melhor: Crie download restrito a /srv/reports, normalize path, bloqueie traversal, limite extensão e cubra com teste negativo.

Java

Log4Shell como lembrete de transitivas

gate: OWASP Dependency-Check
Vulnerável
<dependency>
  <groupId>org.apache.logging.log4j</groupId>
  <artifactId>log4j-core</artifactId>
  <version>2.14.1</version>
</dependency>
Seguro
<dependencyManagement>
  <dependencies>
    <dependency>
      <groupId>org.apache.logging.log4j</groupId>
      <artifactId>log4j-bom</artifactId>
      <version>2.23.1</version>
      <type>pom</type>
      <scope>import</scope>
    </dependency>
  </dependencies>
</dependencyManagement>

Risco: Versão vulnerável ou transitiva não observada pode entrar no build sem alteração direta de código.

Gate: OWASP Dependency-Check, Maven Enforcer, lock/dependency tree e política de atualização.

Prompt fraco: Adicione log4j para logs.

Prompt melhor: Escolha versão mantida, valide CVEs transitivas, gere dependency tree e configure SCA no pipeline.

C#

Autorização quebrada por recurso

gate: Teste com dois usuários
Vulnerável
[Authorize]
public async Task<IActionResult> GetInvoice(Guid id) {
  return Ok(await db.Invoices.FindAsync(id));
}
Seguro
[Authorize]
public async Task<IActionResult> GetInvoice(Guid id) {
  var userId = User.FindFirstValue(ClaimTypes.NameIdentifier);
  var invoice = await db.Invoices.SingleOrDefaultAsync(i => i.Id == id && i.OwnerId == userId);
  return invoice is null ? NotFound() : Ok(invoice);
}

Risco: Usuário autenticado consegue consultar recurso de outro usuário alterando o id.

Gate: Teste com dois usuários, policy-based auth e revisão de BOLA/IDOR.

Prompt fraco: Proteja o endpoint com login.

Prompt melhor: Proteja por usuário dono do recurso, tenant e ação; inclua teste negativo acessando invoice de outro usuário.

CI/CD

Secrets em PR não confiável

gate: Permissões mínimas
Vulnerável
on: pull_request_target
permissions: write-all
steps:
  - uses: actions/checkout@v4
    with: { ref: ${{ github.event.pull_request.head.sha }} }
  - run: npm install && npm test
    env:
      NPM_TOKEN: ${{ secrets.NPM_TOKEN }}
Seguro
on: pull_request
permissions:
  contents: read
steps:
  - uses: actions/checkout@v4
  - run: npm ci --ignore-scripts
  - run: npm test
# publish/deploy roda só em branch protegida e ambiente aprovado

Risco: Código do fork pode rodar com token e permissões elevadas.

Gate: Permissões mínimas, ambientes protegidos, secret scan e revisão de workflows.

Prompt fraco: Faça o PR testar e publicar pacote se passar.

Prompt melhor: Separe PR não confiável de publish; sem secrets em fork, permissões mínimas e approval para release.

Docker

Imagem root e superfície desnecessária

gate: Docker Scout/container scan
Vulnerável
FROM node:latest
WORKDIR /app
COPY . .
RUN npm install
CMD ["npm", "start"]
Seguro
FROM node:22-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --omit=dev --ignore-scripts

FROM node:22-alpine
WORKDIR /app
USER node
COPY --from=deps /app/node_modules ./node_modules
COPY --chown=node:node . .
CMD ["node", "server.js"]

Risco: Tag latest, install não determinístico, scripts de pacote e execução como root.

Gate: Docker Scout/container scan, pinning, usuário não-root e revisão de scripts.

Prompt fraco: Crie Dockerfile simples que rode a app.

Prompt melhor: Crie Dockerfile determinístico, imagem mínima, npm ci --ignore-scripts, usuário não-root e scan de CVEs.

IaC

Bucket público e IAM wildcard

gate: Checkov/tfsec
Vulnerável
resource "aws_s3_bucket_acl" "public" {
  bucket = aws_s3_bucket.data.id
  acl    = "public-read"
}

actions = ["s3:*"]
resources = ["*"]
Seguro
resource "aws_s3_bucket_public_access_block" "data" {
  bucket                  = aws_s3_bucket.data.id
  block_public_acls       = true
  block_public_policy     = true
  ignore_public_acls      = true
  restrict_public_buckets = true
}

actions   = ["s3:GetObject"]
resources = ["${aws_s3_bucket.data.arn}/reports/*"]

Risco: Exposição de dados e privilégio excessivo por conveniência de deploy.

Gate: Checkov/tfsec, revisão de policy, aprovação de exceção e teste de acesso.

Prompt fraco: Faça o bucket funcionar publicamente para o app ler.

Prompt melhor: Bloqueie acesso público, use IAM mínimo por prefixo, gere policy testável e explique exceções se existirem.

06 IA

IA no desenvolvimento: potência com limite

Vibe coding e agentes vieram para ficar. O guia trata IA como parte normal do SDLC, com controles explícitos.

Aceleradores

Onde IA ajuda

  • Explorar alternativas e APIs.
  • Gerar testes negativos e abuse cases.
  • Explicar código legado.
  • Propor refatoração pequena e revisável.
  • Revisar checklist de segurança antes do PR.
Riscos

Onde IA quebra

  • Alucina pacotes e flags de segurança.
  • Remove validação para "simplificar".
  • Segue instrução maliciosa em arquivo do repo.
  • Lê segredo se o escopo permitir.
  • Gera diff grande demais para review real.
Controles

Como operar agentes

  • Workspace isolado e sem acesso ao home.
  • Rede bloqueada por padrão ou allowlist.
  • Approval para dependências, rede e comandos destrutivos.
  • Secrets fora do contexto.
  • Diff pequeno, testes e ameaça residual no resumo.
Agent Permission Lab

Monte o perfil de execução do agente

Ative permissões e observe o risco residual. Um agente produtivo não precisa de acesso total para entregar código revisável.
